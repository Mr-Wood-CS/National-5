# Analysis Worked Example

This page shows one way to work through an analysis question.

!!! note "Teaching Note"
    In a real project, analysis is often an iterative process. Developers may move back and forward between user needs, functional requirements and database design. In this worked example, the steps are separated to make the process easier to follow.

## School Library Example

A secondary school wants to create a database to keep track of its library books, the pupils who borrow them, and the borrowing history.

The database will help school staff manage books and allow pupils to search for and borrow books.

## Step 1: Identify the Users

The users are the people who will use the database.

In this example, the users are:

- Library staff
- Pupils

## Step 2: Identify the End-User Requirements

These are the tasks the users need to complete.

The users need to be able to:

- Search for books by title, author, or genre
- Find out if a book is available or already on loan
- View a list of books a pupil has borrowed
- Check which books are overdue
- Sort books by title or date added

!!! tip "Top Tip"
    End-user requirements are often written as "The user needs to be able to...".

## Step 3: Choose Tables and Fields

The database needs tables and fields to store the information.

For this example, the database needs three linked tables:

1. __Book__ – stores details of books in the library
2. __Pupil__ – stores details of pupils who borrow books
3. __Loan__ – stores details of which books have been borrowed and returned

Each table needs a __primary key__ to uniquely identify each record. The Loan table will also need __foreign keys__ to link each loan to the correct book and pupil.

## Step 4: Identify the Functional Requirements

These describe what the database must store, link and do.

The database must be able to:

- Search books by title, author, or genre **(uses the Title, Author and Genre fields)**
- Sort results alphabetically or by date **(uses fields such as Title or Date Borrowed)**
- Check if a book is available **(uses the Available field or the Loan table)**
- List all books borrowed by a pupil **(uses links between the Pupil, Loan and Book tables)**
- Identify overdue books **(uses the Date Borrowed and Date Returned fields)**

## Step 5: Check the Link Between Requirements

Use one end-user requirement and check that the functional requirements support it.

!!! example "Check the Connection"
    End-user requirement: Library staff need to identify overdue books.

    Functional requirements: The Loan table stores Date Borrowed and Date Returned. The database can use these dates to find overdue loans.

!!! summary "Summary"

    Start with the users, then list what they need to do. After that, identify the tables, fields, keys and actions the database needs to support those tasks.
